# 1. The Base Blueprint (Lightweight Linux with Python 3.11)
FROM python:3.11-slim
# What it does: Every Dockerfile in the world must start with FROM. It tells Docker what pre-existing snapshot to use as the foundation.
# In plain English: "Go to the official Docker server, download a blank Linux computer that has Python 3.11 installed, and start my recipe from there."
# ---------------------------------------------------------------------------
 
# 2. The Math/C++ Trap Fix (Install system dependencies for FAISS)
# We update the Linux package manager and install the C++ compilers
RUN apt-get update && apt-get install -y \
    build-essential \
    libopenblas-dev \
    libomp-dev \
    && rm -rf /var/lib/apt/lists/*

# ------ RUN (The Keyboard Typist - During Build) ----------
# What it does: Imagine you are sitting at the Linux terminal typing commands. RUN executes standard terminal commands while the image is being built.
# In plain English: "Hey Linux, use your package manager (apt-get) to download C++ compilers. Then use Python's package manager (pip) to download FastAPI and FAISS."

# ***apt-get update: This refreshes the Linux app store index so it knows what the newest packages are.
# ***-y: This stands for "Yes". When you install software on Linux, it usually pops up a prompt saying: "This will take 50MB of space, do you want to continue? [Y/n]". 
# Because Docker builds automatically without a human watching it, it doesn't have a keyboard to type "Y". If you omit -y, the build will freeze forever waiting for an input. This flag forces an automatic "Yes".
# ***&& rm -rf /var/lib/apt/lists/*: This is a cleanup trick. When Linux downloads installation files, it stores them in a temporary folder. Once the installation is finished,
# we don't need those installer files anymore. This deletes them to keep our final Docker image as small and lightweight as possible.
# 2. The rm and -rf (The Linux Wrecking Ball)
# In Linux, there is no "Recycle Bin." When you delete something from the terminal, it is gone forever.
# ***rm: Stands for remove. It is the basic command to delete a file.
# ***-r: Stands for recursive. If you try to delete a folder using just rm, Linux will stop you and say, "Wait, there are files inside this folder!" The -r flag tells Linux to violently drill down into the folder and delete every file and sub-folder inside it.
# ***-f: Stands for force. Normally, if you delete system files, Linux gets scared and asks: "Are you sure you want to delete this? [y/n]". The -f flag tells Linux to shut up, not ask any questions, and just destroy the files immediately.
# Putting it together: rm -rf /var/lib/apt/lists/* translates to: "Delete all the junk installation folders in this directory, dig all the way to the bottom, and do not ask me for permission." We do this to keep our Docker image small.
# ---------------------------------------------------------------------------

# 3. Set up our working directory inside the Docker image
WORKDIR /app
# 3. WORKDIR (The Folder Setup)
# Code: WORKDIR /app
# What it does: It creates a directory inside your Linux container and basically types cd /app (change directory).
# In plain English: "Create a folder named app in the root of the Linux hard drive, and make sure all future commands happen inside this folder."
# ---------------------------------------------------------------------------

# 4. Copy the requirements file and install Python libraries
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
# ***--no-cache-dir: When you run pip install normally, Python downloads the library files and keeps a backup copy (a cache) on your hard drive just in case you want to reinstall them later.
# In a Docker container, we are only ever going to build this image once. Keeping a backup copy inside the frozen image is completely useless and wastes hundreds of megabytes. 
# This flag tells Python: "Install the libraries, and delete the installer downloads immediately."

# 4. COPY (The Teleporter)
# Code: COPY requirements.txt .
# Code: COPY . .
# What it does: It takes files from your physical Windows laptop and copies them inside the Linux container. The syntax is always COPY [source] [destination]. The . just means "right here."
# In plain English: "Take my code from Windows and teleport it right into the /app folder we just made inside Linux."

# 1. The -r in pip install -r
# What it means: Read from a file.
# Why we use it: Normally, you install a package one by one: pip install fastapi. But if you want to install 10 packages at once, typing them all out is annoying. 
# The -r flag tells pip: "Don't look for a single package name. Look at this text file (requirements.txt), read it line by line, and install every single thing listed inside it."
# ---------------------------------------------------------------------------

# 5. The Machine Learning Pre-bake
# We run a tiny Python command to force the container to download 
# the BGE model directly into the frozen image, saving startup time.
RUN python -c "from sentence_transformers import SentenceTransformer; SentenceTransformer('BAAI/bge-small-en-v1.5')"
# ***-c: This stands for "Command". Normally, if you just type python in a terminal, it opens an interactive Python shell where you type code line-by-line. 
# The -c flag tells the system: "Don't open an interactive terminal. Just execute the code inside these quotation marks right now like a script, and then close." 
# We do this during the build process to force the code to download the ML model weights from the internet and bake them directly into the hard drive layers of the Docker image.
# ---------------------------------------------------------------------------

# 6. Copy all the code into the container
COPY . .
# It means "copy every single file and folder." But let's break down exactly why it uses two dots.
# In terminal language, a single dot . means "The exact folder I am standing in right now."
# The First Dot (Source): Represents your Windows folder. "Grab everything in the folder I am currently standing in on my laptop."
# The Second Dot (Destination): Represents the Linux folder. "Paste all of it into the folder I am currently standing in inside the Linux container."
# (Because we used WORKDIR /app earlier, the container is currently standing in /app).

# The Danger of COPY . .:
# Because it copies everything, it will also copy your massive .venv (virtual environment) folder and your __pycache__ folders from Windows into Linux. Linux will get very confused by Windows environment files.
# To fix this, we create a tiny text file next to your Dockerfile called .dockerignore (it works exactly like .gitignore). You just put .venv inside it, and Docker will skip it when it sees the two dots.
# ---------------------------------------------------------------------------

# 7. The Execution Command (What happens when the container starts)
CMD ["uvicorn", "gateway_v2:app", "--host", "0.0.0.0", "--port", "8000"]

# 5. CMD (The Big Red Button - During Launch)
# Code: CMD ["uvicorn", "gateway_v2:app", "--host", "0.0.0.0", "--port", "8000"]
# What it does: This is the only command in the file that does not execute during the build process. It is the instruction manual for the container.
# In plain English: "When someone finally turns this container on in the future, automatically run this exact Uvicorn command to start the web server."

# --host 0.0.0.0: This is the most critical filling in the entire file. Normally, when you run Uvicorn on your laptop, it hosts on 127.0.0.1 (Localhost). 127.0.0.1 means "Only talk to the machine I am currently sitting inside."
# Inside Docker, the container is its own isolated computer. If Uvicorn binds to 127.0.0.1, it will only accept requests coming from inside the tiny container itself. Your physical Windows laptop will be completely blocked out. 
# By setting the host to 0.0.0.0, you are telling the container: "Listen to any network traffic coming from the outside world, including requests from my Windows operating system."

# 4. The 8000 in the CMD
# Think of an IP address (like 0.0.0.0 or 127.0.0.1) as the physical street address of an apartment building. But an apartment building has hundreds of doors. Which door does the data knock on?

# A Port is the door number.
# Door 80 is usually for standard websites (HTTP).
# Door 443 is for secure websites (HTTPS).
# Uvicorn and FastAPI, by default, always set up their shop behind door 8000.
# When we write --port 8000 in the Dockerfile, we are explicitly forcing the Uvicorn server to listen to door number 8000 inside the container. Later, when we run the container, 
# we will tell Docker to connect door 8000 on your Windows laptop to door 8000 on the Linux container, creating a perfect tunnel for your web traffic.
